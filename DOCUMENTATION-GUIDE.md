# Documentation Guide

> 本工作区的文档编写标准和最佳实践。

## 📋 文档层级

每个项目应包含以下文档（按优先级排序）：

| 优先级 | 文件 | 作用 | 必须性 |
|--------|------|------|--------|
| P0 | `README.md` | 项目概览、快速开始 | ✅ 必须 |
| P1 | `API.md` | 完整 API 参考 | 推荐（库/框架项目） |
| P1 | `TUTORIAL.md` | 逐步教程 | 推荐（面向用户的项目） |
| P2 | `CONTRIBUTING.md` | 贡献指南 | 推荐（开源项目） |
| P2 | `CHANGELOG.md` | 版本变更记录 | 推荐（有版本管理的项目） |
| P3 | `ARCHITECTURE.md` | 架构设计文档 | 可选（复杂系统） |
| P3 | `FAQ.md` | 常见问题 | 可选 |

## ✍️ 写作风格

### 语言选择

- **面向国际用户的项目**：英文为主，关键概念可附中文注释
- **内部工具/实验性项目**：中文为主
- **代码注释**：英文（与代码一致）

### README.md 结构模板

```markdown
# 项目名称 🏷️

> 一句话描述项目

## 概述
2-3 段说明项目做什么、为什么存在、解决了什么问题。

## 快速开始
最短路径让用户跑起来。不要前置大量概念解释。

\```bash
pip install xxx
\```

\```python
from xxx import Yyy
result = Yyy().run("hello")
\```

## 安装
详细安装步骤、环境要求、可选依赖。

## 使用示例
3-5 个典型用例，从简单到复杂。

## API 参考
核心类和方法的简要说明。详细版本放 API.md。

## 项目结构
目录树 + 每个目录/文件的简要说明。

## 开发
如何搭建开发环境、运行测试、构建。

## 许可证
```

### 关键原则

1. **代码优先** — 能用代码展示就不用文字描述
2. **渐进式** — 快速开始 → 详细文档 → 高级用法
3. **可运行** — 所有代码示例必须能直接复制运行
4. **解释 Why** — 不仅说明做什么，还要说明为什么这样做

## 🔍 质量检查清单

### README 检查

- [ ] 一句话说明项目是什么
- [ ] 5 分钟内能跑起来（快速开始）
- [ ] 有代码示例且可运行
- [ ] 项目结构清晰
- [ ] 安装步骤完整
- [ ] 包含许可证信息
- [ ] 没有明显的拼写/语法错误

### API 文档检查

- [ ] 每个公共类/函数都有文档
- [ ] 参数类型和返回值类型明确
- [ ] 有使用示例
- [ ] 异常和错误情况已说明
- [ ] 破坏性变更有标注

### 教程检查

- [ ] 从零开始，不假设前置知识
- [ ] 每一步都可以独立验证
- [ ] 有完整的端到端示例
- [ ] 包含常见问题排查

## 📐 格式规范

### 中文排版

- 中英文之间加空格：`使用 OpenClaw 框架`
- 中文与数字之间加空格：`版本 3.0`
- 使用全角标点
- 专有名词保持原文：`Agent`、`Mesh`、`Tool`

### 代码块

- 指定语言：` ```python ` 而不是 ` ``` `
- 代码块内用英文注释（与代码一致）
- 超过 30 行的代码考虑拆分或加行号说明

### 链接

- 优先用相对链接：`[API 文档](API.md)` 而不是绝对 URL
- 外部链接用完整 URL
- 链接文字要有描述性，避免"点击这里"

## 🔄 文档维护

### 何时更新文档

| 触发事件 | 更新内容 |
|----------|---------|
| 新增公共 API | API.md + README 示例 |
| 修改公共 API | API.md + CHANGELOG.md |
| 新增功能 | README + TUTORIAL |
| 修改安装步骤 | README 安装章节 |
| 修复 Bug | FAQ（如果是常见问题） |
| 架构变更 | ARCHITECTURE.md |

### 过期标记

不确定内容是否准确时，添加标记：

```markdown
> ⚠️ **注意**: 此部分可能已过期，最后验证于 2026-03-01
```
