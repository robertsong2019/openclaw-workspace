# Agent Cluster: Cathedral vs Bazaar

> 从单 Agent 到 Agent 集群的演化，借鉴《大教堂与集市》的组织模式

## 两种模型

### 大教堂模式（传统企业分工）
- 自顶向下，主 Agent 分配任务层层下达
- 角色固定，流程严格
- 优点：可控、可预测、容易调试
- 缺点：单点瓶颈、适应性差

### 集市模式（开源社区）
- 自底向上，Agent 自己发现问题认领任务
- 角色流动，粗略共识 + 可执行代码
- 优点：去中心化、自愈能力强、规模扩展性好
- 缺点：可能重复劳动、一致性难保证

## 推荐：混合三层架构

1. **协调层（集市广场）** — 共享黑板/任务板，Agent 自由发布和认领
2. **团队层（项目组）** — 2-4 个 Agent 按项目固定分组协作
3. **个体层（Linus 法则）** — 任何 Agent 可审查任何输出

## 集市适合的场景
- 代码审查（多 Agent 审查同一 PR）
- 知识库维护（任何 Agent 发现冲突可修复）
- Bug 发现（足够多 Agent 看过，所有 bug 无所遁形）

## 大教堂适合的场景
- 部署流水线（严格顺序执行）
- 安全敏感操作（明确授权和审计链）
- 资源争抢（数据库写入、API 调用）

## 演进路线
- 短期：大教堂 + 少量集市元素（任务板 + 交叉审查）
- 长期：集市为主，大教堂为辅（Agent 自治程度随能力提升）
