# LLM Cache Strategy

> 先给地图再找路 — 两级缓存降低噪音提高精度

## Karpathy 的检索方式

不做传统语义搜索（向量相似度），而是纯靠 LLM 上下文窗口 + 文本索引：

1. **L1 缓存（快）** — index.md，全局目录，几K大小，秒加载
2. **L2 缓存（按需）** — 具体 Wiki 页面，根据 L1 判断后精读
3. **L3（兜底）** — raw/ 原始资料，很少需要直接访问

## 为什么中等规模够用

100 来源、几百页面 → index.md 几千字 → 完全塞进上下文窗口。LLM 读完目录就能精确判断该去哪几个页面。

## 与传统方案对比

| | 向量检索（RAG） | LLM 索引定位（Wiki） |
|---|---|---|
| 依赖 | 向量数据库、embedding 模型 | 纯文本、LLM 语义理解 |
| 精度 | 受 embedding 质量影响 | LLM 直接理解语义 |
| 成本 | 需要基础设施 | 零额外成本 |
| 规模上限 | 百万级 | 几百~几千页面 |
